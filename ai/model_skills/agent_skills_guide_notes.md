# Agent Skills 终极指南：最值得关注的内容

来源：[少数派《Agent Skills 终极指南：入门、精通、预测》](https://sspai.com/post/105230)  
整理日期：2026-06-09

## 一句话结论

Skills 值得关注的核心，不是“又多了一种提示词写法”，而是它把 Agent 的工作方法、领域知识、脚本、模板和素材打包成可复用的能力包。它让通用 Agent 可以低成本变成垂直 Agent：人负责沉淀方法，模型负责理解、规划和执行。

## 最值得关注的 8 个要点

### 1. Skill 是 Agent 的能力扩展包

文章把 Skill 类比为“工作交接 SOP 大礼包”。一个 Skill 可以包含：

- 任务执行 SOP 与背景知识
- 工具使用说明
- 模板、素材、历史案例
- 常见问题、规范和解决方案
- 可直接运行的脚本

因此，Skill 不只是 prompt，而是“指令 + 资源 + 可执行能力”的组合。

### 2. Skill 与 MCP 的区别

MCP 更像统一的外部工具、数据和服务调用协议，解决“Agent 如何连接外部能力”的问题。

Skill 更像任务方法封装，解决“Agent 如何稳定完成一类工作”的问题。它可以告诉 Agent 什么时候做什么、按什么流程做、用哪些脚本或材料做。

可以粗略理解为：

- MCP：连接工具和服务
- Skill：封装任务逻辑、知识和操作方法

两者可以互补：Skill 可以教 Agent 如何使用某个 MCP 服务。

### 3. 标准结构里只有 SKILL.md 是必需的

文章强调，Skill 的标准形态很轻。一个最小 Skill 可以只有 `SKILL.md`，复杂 Skill 再加入脚本、参考材料和资源。

典型目录可以这样设计：

```text
my-skill/
  SKILL.md
  scripts/
  reference/
  assets/
  templates/
```

其中：

- `SKILL.md`：核心说明，包含元数据、触发条件、流程、约束和最佳实践
- `scripts/`：可执行脚本，用于提高稳定性和减少重复开发
- `reference/`：领域知识、规范、说明文档
- `assets/`：图片、Logo、素材、示例文件
- `templates/`：输出模板、文档模板、提示模板

### 4. 渐进式披露是 Skill 的关键机制

文章最重要的技术点之一是“渐进式披露”：Skill 并不会把所有内容一次性塞进上下文，而是分层加载。

```text
Level 1: 元数据
  name + description，始终可见，用于判断是否触发 Skill

Level 2: SKILL.md 正文
  触发 Skill 后读取，提供主要工作流程和约束

Level 3: 子文档、脚本、参考材料、资产
  需要时才读取或执行
```

这意味着，一个 Agent 可以安装很多 Skills，但平时只加载少量元信息。真正需要执行任务时，再按需读取具体文档、脚本和资源。

### 5. Skill 的价值在于把领域经验产品化

文章最值得学习的观点是：非技术领域专家可以把自己的专业流程写成文档，让 Agent 获得垂直能力。

比如：

- 写作 Skill：沉淀选题、资料处理、大纲、正文风格
- 品牌设计 Skill：沉淀品牌色、字体、Logo、版式规范
- 数据分析 Skill：沉淀指标定义、异常值处理、图表规范
- 项目复盘 Skill：沉淀文件扫描、结论提炼、输出模板

这种方式降低了垂直 Agent 的验证成本。做一个内部小工具或 MVP，不一定先开发完整产品，可以先做 Skill。

### 6. Skill 比传统 Workflow 更能处理边缘情况

传统 Workflow 通常要求你提前定义入口、字段、格式和分支。一旦用户输入不符合预期，就容易卡住。

Skill + Agent 的模式更灵活，因为 Agent 可以基于 Skill 的目标和约束自行判断：

- 用户上传格式不对时，是否先转换
- 文档缺标题时，是否自动补标题
- 数据结构不完整时，是否先清洗或追问
- 某个步骤缺工具时，是否临时写脚本或调用其他 Skill

这也是 Skill 适合复杂知识工作的原因：它不只是固定流程，而是“有指导的智能执行”。

### 7. 多个 Skills 可以组合使用

文章指出，Skills 本质上是上下文工程，可以在一次任务中联用。

例如做产品分析报告：

- 网页抓取 Skill：收集竞品信息
- PDF Skill：提取用户反馈
- 数据分析 Skill：生成图表和结论
- 品牌规范 Skill + PPTX Skill：制作符合品牌规范的汇报材料

这点对学习很重要：不要只把 Skill 想成单一功能。更有价值的方向，是把复杂工作拆成多个可组合能力。

### 8. 判断是否该做 Skill 的 3 个信号

文章给出的判断标准很实用：

1. 你反复向 AI 解释同一件事
2. 某类任务需要特定知识、模板或材料才能做好
3. 一个任务需要多个流程协同完成

只要命中其中一个，就值得考虑把它沉淀成 Skill。

## 学习 Skills 开发时应优先掌握什么

### 第一优先级：写好 Skill 元数据

`description` 决定 Agent 是否会自动触发 Skill。它应该写清楚：

- 这个 Skill 解决什么任务
- 什么情况下应该使用
- 输入通常是什么
- 输出应该是什么

示例：

```yaml
---
name: article-summary
description: 用于阅读长文章并输出结构化中文学习笔记。当用户提供文章链接、网页内容、PDF 或长文本，并希望提炼观点、方法、行动建议或复习卡片时使用。
---
```

### 第二优先级：把流程写成可执行步骤

不要只写“请总结文章”，而要写清楚 Agent 的工作方式：

```md
## Workflow

1. 先识别文章主题、作者立场和目标读者。
2. 提取核心判断，不按原文顺序机械摘要。
3. 区分概念、方法、案例、争议点和行动建议。
4. 输出 Markdown，包含来源、重点、实践清单和待验证问题。
```

### 第三优先级：把稳定任务交给脚本

如果某个步骤需要重复执行、格式要求严格、容易出错，就适合放进 `scripts/`。

例如：

- 转换文件格式
- 批量读取目录
- 生成固定结构的 Markdown
- 校验输出字段
- 处理表格、PDF、PPTX

文章里的关键提醒是：脚本本身不会进入上下文，只有运行结果进入上下文，所以脚本能降低 token 消耗，也能提升稳定性。

### 第四优先级：把大材料放到 reference 或 assets

不要把所有资料都塞进 `SKILL.md`。`SKILL.md` 应该短而清晰，大量材料放到按需读取的目录中。

适合放到 `reference/` 的内容：

- 术语表
- 公司规范
- 写作风格说明
- 业务流程文档
- 评审标准

适合放到 `assets/` 的内容：

- Logo
- 图片素材
- 示例文件
- 设计资源

## 我的实践建议

如果你正在学习 Skills，建议先做一个“小但完整”的 Skill，而不是一开始做复杂 Agent。

可以从这个方向开始：

```text
article-note-skill/
  SKILL.md
  templates/
    learning-note.md
```

目标：给任意文章链接或文本，输出固定格式的中文学习笔记。

这个练习刚好覆盖 Skill 开发的核心能力：

- 写触发描述
- 写工作流程
- 写输出格式
- 使用模板
- 处理边界情况

## 可直接复用的 Skill 设计检查清单

- `name` 是否短、清晰、唯一
- `description` 是否写清楚触发场景
- `SKILL.md` 是否说明输入、输出和工作流程
- 是否区分了必须遵守的规则和可灵活处理的建议
- 是否把大段参考材料移到 `reference/`
- 是否把重复、机械、容易出错的任务脚本化
- 是否提供了输出模板或示例
- 是否说明了遇到缺失信息时应该追问、假设还是继续
- 是否避免把所有内容一次性塞进主文档

## 这篇文章对我最有启发的地方

Skill 的真正价值不是“让 AI 多记住一些规则”，而是把人类专家的工作方法变成 Agent 可执行、可组合、可复用的能力模块。

从产品角度看，它降低了垂直 Agent 的验证成本。

从个人效率角度看，它适合沉淀那些“我每次都要教 AI 一遍”的工作。

从工程角度看，它把 prompt、脚本、知识库、模板和工具说明组织成了一个更轻量的 Agent 应用形态。

## 后续学习路线

1. 先手写一个只有 `SKILL.md` 的最小 Skill。
2. 给它补一个输出模板，练习 `templates/`。
3. 加入一份参考材料，练习 `reference/` 的按需读取。
4. 加入一个简单脚本，练习 `scripts/` 的稳定执行。
5. 最后再尝试组合多个 Skills 完成一个复杂任务。
